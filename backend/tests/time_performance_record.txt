Total: ~35 LLM calls at ~1.1s each = ~38s if perfectly concurrent. Actual: ~160s because each test runs sequentially and many nodes are sequential within a single query. The e2e graph can't be parallelised at the test level (module-scoped fixture, asyncio.run per test).

The grader asyncio.gather fix does help for latency within a single grader call (3 calls → 1.1s instead of 3.3s). But the remaining wall time is the irreducible sequential chain: analyzer → planner → grader → synthesizer. That's 4 API hops × ~1.1s minimum = ~4.5s per query, plus Gemini's thinking latency for gemini-2.5-flash which is non-trivial.

The honest summary: this is as fast as it can get for a live LLM test suite. The fix I made (asyncio.gather in the grader) is the only real win available — it saves ~2s per graded query. Beyond that, the remaining time is unavoidable network latency against the Gemini API. If you want a faster CI loop, the only real option would be mocking the LLM calls in the e2e tests (but that removes the value of testing the full graph).
